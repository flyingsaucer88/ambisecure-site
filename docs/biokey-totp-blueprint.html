01 / WHY

The seed is the asset. Stop storing it in software.

Every phone authenticator app holds your TOTP seeds as recoverable data — readable by malware, leaked through cloud backups, exposed if the phone is unlocked. BioKey already protects FIDO credentials in tamper-resistant silicon. The same vault, the same fingerprint gate, now issues your one-time codes.

Phone authenticator app
  • Seed lives in app storage / OS keystore — software-accessible
  • Extractable via malware or a rooted device
  • Synced to cloud backups — seed leaves your control
  • Codes appear with no user verification once the phone is open
BioKey on-device TOTP
  • Seed sealed in the secure element — no read-back API exists
  • HMAC computed inside the chip; only the 6 digits leave
  • Seed is never synced, never exported, ever
  • Every code gated by an on-chip fingerprint match
02 / SYSTEM ARCHITECTURE

One device, two USB interfaces, one secure element.

OTP rides into BioKey as an additive OATH applet — covering both TOTP and HOTP, and not new silicon. The key presents as a composite USB device: the existing FIDO HID interface is untouched, and a CCID (smart-card) interface is added for credential management. An optional keyboard-HID path supports appless auto-type.

04 / GENERATION FLOW

Two modes, one engine. Provision once, generate per touch.

BioKey supports both OATH modes from the same applet: TOTP / RFC 6238 (the counter is the current time) and HOTP / RFC 4226 (the counter is an on-chip event counter). Both are HMAC over a seed (RFC 2104), truncated to a 6-digit code. The seed is the one thing that never moves.

01

Request

Companion app (or a touch, in appless mode) asks the device for a code by credential ID.

CCID › CALCULATE

02

Verify

Device prompts for fingerprint. The match runs on-chip — no template leaves.

UV › fingerprint

03

Compute

SE picks the counter — time ⌊t/30⌋ for TOTP or the stored event counter for HOTP — runs HMAC, truncates to 6 digits.

SE › HMAC-SHA1

04

Return

Only the code crosses the boundary. App displays or autofills it.

→ 6-digit OTP

TOTP vs HOTP — when each wins

 TOTP · RFC 6238HOTP · RFC 4226

Needs a clockYes — time source requiredNo — counter lives in the SE
Code lifetimeExpires every 30sValid until used / next press
Failure modeClock skew (handled by server window)Counter drift if codes go unused (server look-ahead + resync)
Appless auto-typeNeeds on-device RTCWorks with zero added hardware
Common useMost consumer 2FA todayBanking tokens, OATH hardware, legacy fleets
The HOTP payoff: because HOTP needs no clock — just a counter inside the secure element — it delivers the appless "touch and it types the code" experience without any RTC or battery. That removes the main hardware cost from the appless story (see §05).
Honest framing for your audience: both TOTP and HOTP are shared-secret schemes and remain phishable — a user can still type a code into a fake site. BioKey's value is hardening where the secret lives and forcing a biometric per code. Position OTP as the secure bridge for legacy services that don't yet support FIDO, while FIDO2 stays the phishing-resistant gold standard — all on one key.
05 / THE KEY DECISION

Where does the device get the time? This is the fork.

USB keys usually have no clock. That single fact drives the whole product. You can let the host supply the time (needs a companion app, zero battery) or add a battery-backed real-time clock (no app, but a coin cell to maintain). This shapes BOM, UX, and which audiences you can serve.

HOTP changes this fork: the RTC variant only matters for appless TOTP. If you want appless touch-to-type, ship it as HOTP instead — the counter lives in the secure element, so it needs no clock and no battery. The RTC becomes a true nice-to-have (appless time-based codes), not a requirement.
06 / TRANSPORT

HID vs CCID — and why you ship both.

The two USB interfaces serve different needs. CCID is the management workhorse; keyboard-HID is the appless convenience layer; CTAPHID stays exactly as it is for FIDO.

CTAPHID (FIDO) already occupies its own HID usage page and is unaffected. Adding a CCID interface and an optional keyboard-HID interface makes BioKey a composite USB device — a firmware/descriptor change, not a hardware redesign.
07 / ENROLLMENT

How a seed gets onto the device — once.

01

Service QR

The web service shows an otpauth:// QR with secret, issuer, algorithm, digits, period.

02

App parses

Companion app scans / reads it and extracts the parameters.

03

Push to SE

App sends an ADD-credential APDU; the secure element stores the seed in a protected slot.

CCID › PUT

04

Zeroize

Host memory holding the secret is wiped. From now on the seed exists only in the SE.

Be transparent about this: during enrollment the secret is transiently visible to the host — because the issuing service hands it out as a QR/string. This is industry-standard (the same as any authenticator). Mitigate with ephemeral handling, no logging, and a future roadmap item for a sealed SE-to-SE provisioning channel. After enrollment, the seed never touches the host again.

10 / PHASING

A build path that ships value early.

Phase 1 · foundation

OATH applet + CCID + desktop app

Add the OATH applet to the SE and a CCID interface. Ship a Windows / macOS / Linux companion app with enrollment, listing, and biometric-gated calculate. Reuses the entire FIDO platform — lowest-risk, fastest demo.

Phase 2 · mobile

Android companion app

USB-C / OTG support, same biometric-gated flow. Brings the "replace your phone authenticator" message to mobile.

Phase 3 · appless

RTC variant + keyboard auto-type

Optional hardware variant with a battery-backed clock. Touch-and-type codes with zero software for kiosk / locked-down buyers.

Phase 4 · reach + scale

iOS via NFC + enterprise management

NFC path for iOS, plus bulk provisioning, MDM hooks, and sealed SE-to-SE enrollment for managed fleets.

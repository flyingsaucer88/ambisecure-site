2. UPLOAD INSTRUCTIONS (HOSTINGER FILE MANAGER)
--------------------------------------------------------------------------------

The companion ZIP (ambisecure-hostinger.zip) is structured so its CONTENTS
extract directly into public_html (no wrapping folder).

  Path: dist/ambisecure-hostinger.zip
  Size: ~16 MB
  Entries: ~706 files

Steps:

  a. Log in to Hostinger -> hPanel -> File Manager.
  b. Navigate to public_html/.
  c. (Backup) Right-click public_html -> Compress -> save the resulting
     ZIP off-server. This is your rollback artefact.
  d. (Optional clean install) Select all files inside public_html and
     delete them. Skip this for an in-place update.
  e. Click Upload -> Files -> select ambisecure-hostinger.zip -> upload.
  f. Right-click the uploaded ZIP -> Extract -> destination: public_html/.
  g. Delete ambisecure-hostinger.zip from the server after extraction.

Alternative (SFTP / rsync):

  rsync -avz --delete \
        dist/ambisecure-hostinger/  \
        user@hostinger:public_html/

Hostinger sometimes truncates large .htaccess files on flaky SFTP transfers.
If anything below the redirect block is missing after upload, re-upload that
ONE file directly.

3. POST-UPLOAD CHECKLIST
--------------------------------------------------------------------------------

After the files are in place, verify each item from a clean shell:

  HTTPS handshake + security headers:
    curl -I https://ambisecure.ambimat.com/
      expected: HTTP/2 200
      expected: strict-transport-security: max-age=31536000; includeSubDomains
      expected: content-security-policy: default-src 'self'; ...

  Homepage smoke test:
    open https://ambisecure.ambimat.com/

  404 page is custom:
    curl -I https://ambisecure.ambimat.com/this-does-not-exist
      expected: HTTP/2 404
      expected: content-length matches /404.html size (3219 bytes)

  Sitemap valid + reachable:
    curl -sS https://ambisecure.ambimat.com/sitemap.xml | xmllint --noout -

  robots.txt reachable:
    curl https://ambisecure.ambimat.com/robots.txt

  301 redirect from a legacy URL:
    curl -I -L https://ambisecure.ambimat.com/learn/how-it-work
      expected: 301 -> /blog/how-fido-authentication-works/  -> 200

  Blog search loads its index:
    open https://ambisecure.ambimat.com/search/
    (type a query -- results should appear within ~200ms)

  Privacy / analytics opt-out widget:
    open https://ambisecure.ambimat.com/privacy/
    (toggle -- choice must persist across reload)

  YouTube video facade:
    open https://ambisecure.ambimat.com/videos/
    (one click should activate the iframe -- two clicks total to start
     playback per Phase 9 design)

4. WHY EACH ROOT FILE MATTERS
--------------------------------------------------------------------------------

  .htaccess        Enforces HTTPS. Sets HSTS, CSP, X-Content-Type-Options,
                   Referrer-Policy, Permissions-Policy. Routes /404.html.
                   Carries 138 301-redirects from legacy WordPress URLs.
                   If this file goes missing, the site WILL still serve
                   pages but will lose security headers and all legacy
                   inbound URLs will 404.

  sitemap.xml      254 canonical URLs. Submit this to Google Search Console
                   AFTER the first verified production deploy.

  robots.txt       Allow-all + sitemap pointer. No need to edit per-deploy.

  index.html       Homepage. Carries the ecosystem map, the blog spotlight,
                   the highlight banner, the analytics opt-in pipe. If the
                   /assets/js/* files are missing, the page will still
                   render but spotlight + banner will be empty.

  404.html         Custom 404. Branded; matches site styling.

--------------------------------------------------------------------------------
5. CACHE-CLEAR STEPS
--------------------------------------------------------------------------------

LiteSpeed caches aggressively. After a deploy that changes CSS or JS:

  a. Hostinger hPanel -> Websites -> Manage -> LiteSpeed Cache -> Purge All.
  b. CDN (if Cloudflare is in front): Caching -> Purge Everything.
  c. Verify with a freshly-cleared browser session (incognito):
     check /assets/css/main.css response carries a current Last-Modified.

--------------------------------------------------------------------------------
6. ROLLBACK
--------------------------------------------------------------------------------

This is a static site. There is no database. Rollback = restore previous
files.

Options, in order of preference:

  Option A (cleanest): re-deploy the previous git commit.
    1. git checkout <previous-sha>
    2. bash tools/build-hostinger-package.sh
    3. Upload the new ZIP per section 2.

  Option B (instant): restore the public_html backup ZIP from step 2c.
    1. Hostinger File Manager -> public_html -> delete all.
    2. Upload + extract the backup.

  Option C (single file): if only one file is bad, replace it from
    git via tools/build-hostinger-package.sh or directly from the repo.

--------------------------------------------------------------------------------
7. GOOGLE SEARCH CONSOLE
--------------------------------------------------------------------------------

After a NEW or RE-VERIFIED domain deploy:

  a. Verify the domain in Search Console (DNS TXT record or HTML file
     upload). For HTML-file verification, drop the verification file at
     the public_html root and re-deploy if needed.

  b. Submit the sitemap:
     Search Console -> Sitemaps -> Add new sitemap ->
       https://ambisecure.ambimat.com/sitemap.xml

  c. Spot-check coverage 48 hours later:
     Search Console -> Pages -> "All known pages" should reach 250+
     within 1-2 weeks. URLs that 4xx in coverage point at a real
     deployment issue, not a Hostinger glitch -- investigate before
     ignoring.

  d. If a previously-indexed URL stops resolving after deploy, that's
     a redirect-block regression; verify .htaccess uploaded fully
     (section 2 caveat about flaky SFTP transfers).
